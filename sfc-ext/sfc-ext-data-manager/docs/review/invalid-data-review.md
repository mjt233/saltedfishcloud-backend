# 失效数据识别与修复 - 业务流程评审

## Context

评审 `sfc-ext-data-manager` 插件中"失效数据识别与修复"功能的业务流程设计，重点检查流程闭环性、边界条件处理和数据一致性。当前该模块处于纯设计文档阶段，尚无代码实现，正是评审的最佳时机。

设计文档来源：

- `sfc-ext/sfc-ext-data-manager/README.md`
- `sfc-ext/sfc-ext-data-manager/docs/invalid-data-detect.md`

---

## 评审结论

业务流程主干（检测 -> 浏览 -> 修复/认领/丢弃）基本清晰，但存在 **7个流程断点** 和 **4个数据一致性风险**，需要在实现前补充设计。

---

## 一、流程断点（流程不闭环）

### 1. 多次检测的结果管理 [高]

**现状**：设计仅描述"扫描并保存明细"，未说明多次检测的关系。

**问题**：

- 第一次检测出10条失效数据，修复了3条后再次检测，结果是全量覆盖还是增量追加？
- 如果全量覆盖，已修复但未手动确认的记录会丢失上下文
- 如果增量追加，已丢弃/认领的旧记录会重复出现

**建议**：检测结果应全量替换。每次检测开始时清除上一轮未处理（非已修复/已认领/已丢弃）的检测结果，检测完成后再写入新结果。需要定义失效数据记录的状态机：
`待处理 -> 已修复/已认领/已丢弃`。

> 意见：采纳建议，全量替换，每次检测开始时清除上一轮未处理（非已修复/已认领/已丢弃）的检测结果，检测完成后再写入新结果。需要定义失效数据记录的状态机：
`待处理 -> 已修复/已认领/已丢弃`。

### 2. 修复后失效记录的状态更新 [高]

**现状**：README描述了修复动作（重建文件记录、删除失效记录），但未说明修复操作完成后失效数据记录本身如何变化。

**问题**：

- 快速修复成功后，对应的失效数据记录是删除还是标记状态？
- 如果标记状态，下次检测时如何避免重复识别已修复的数据？

**建议**：修复/认领/丢弃操作完成后，将失效数据记录状态标记为已处理（而非直接删除），保留操作历史。再次检测时跳过已处理的记录。这需要在失效数据实体中增加
`status` 字段。

> 意见：快速修复成功后，下次就不会被检测到了，因为已经被修复了就不是失效数据了，文件记录 和 物理存储已重新完成关联。

### 3. UNIQUE模式认领路径构建 [中]

**现状**：认领流程已明确仅针对UNIQUE模式下失效物理存储。RAW模式通过快速修复重建文件记录，无需认领。

**问题**：

- UNIQUE模式下认领时，重建的文件记录存储在哪个目录结构下？UNIQUE模式不区分公共/用户网盘的物理路径，文件记录的逻辑路径如何构建？
- 认领到"公共网盘"时，文件记录的uid=0，物理存储路径如何映射？

**建议**：明确UNIQUE模式下认领创建的文件记录的存储路径规则。例如：认领到"我的网盘"时，文件记录路径为 `/用户的网盘根目录/用户指定的文件名`
；认领到"公共网盘"时，路径为 `/公共网盘根目录/用户指定的文件名`。

> 意见：认领记录通过一个targetUid字段表示保存的目标位置，公共网盘为0，个人网盘为用户uid。文件记录的物理路径根据targetUid、保存路径和文件名构建。

### 4. 认领的文件名冲突处理 [中]

**现状**：认领时"指定文件名、保存路径"，未说明冲突处理。

**问题**：用户指定的文件名+路径如果已存在同名文件，是覆盖、拒绝还是自动重命名？

**建议**：拒绝操作并提示用户文件名已存在，由用户修改后重新认领。这与网盘的常规交互逻辑一致。

> 意见：拒绝操作并提示用户文件名已存在，由用户修改后重新认领。这与网盘的常规交互逻辑一致。

### 5. 检测过程中的操作并发 [中]

**现状**：检测是异步任务，耗时可能较长。

**问题**：

- 检测过程中，管理员能否对上一轮检测结果执行修复/认领/丢弃？
- 如果可以，新检测结果写入时如何与正在进行的操作协调？

**建议**：允许在检测期间操作旧结果。新检测结果写入时只替换"待处理"
状态的记录，不影响正在被操作（已修复/已认领/已丢弃）的记录。如果采用全量替换策略，需要在检测开始时快照当前已处理的记录ID，写入时排除这些ID。

> 意见：限制只能同时有一个进行中的检测 或 文件类型识别任务，检测或识别过程中不允许再次发起检测或文件类型识别任务。

### 6. 失效文件记录快速修复 - 路径冲突 [中]

**现状**：RAW模式下"根据失效文件记录中的文件md5，检查是否存在对应的物理存储文件，若存在则直接重建文件记录"。

**问题**：重建文件记录时使用原路径，但原路径可能已被其他文件占用。

**建议**：重建前检查原路径是否可用。如果已被占用，标记该记录为"修复失败-路径冲突"，由管理员手动处理。

> 意见：采纳建议

### 7. 文件识别与检测结果的关联 [低]

**现状**：README提到"针对需要识别的文件，创建异步任务执行文件识别操作"，但未说明识别结果如何回写。

**问题**：

- 识别任务如何知道要更新哪条失效数据记录？
- 识别结果是直接更新字段还是需要管理员确认？

**建议**：识别任务的参数中包含失效数据记录ID，识别完成后直接更新该记录的 `fileType` 和 `metadata`
字段，无需管理员二次确认（识别结果仅作参考信息）。

> 意见：
> 1. 参考文档 [失效数据检测策略](../invalid-data-detect.md) 中的`## 2. 记录失效数据`
     ，产生的失效数据记录有一个是否为待识别文件字段。识别任务只需要对所有【是否为待识别文件】进行处理即可。
> 2. 识别结果直接更新失效数据记录的 fileType 和 metadata 字段，无需管理员二次确认（识别结果仅作参考信息）。

## 二、数据一致性风险

### 8. 认领后的物理存储引用计数 [高]

**现状**：UNIQUE模式下，同一物理文件（MD5）可被多人多次认领，每次认领创建一条新文件记录。

**问题**：

- 多个文件记录通过MD5引用同一物理存储。删除其中一条文件记录时，是否需要检查是否还有其他引用？
- 如果所有认领的文件记录都被删除，物理存储是否应自动回到失效状态？
- UNIQUE模式下物理存储本身没有"文件记录"的概念（只有MD5索引），认领后失效数据记录的状态如何变化？

**建议**：

- UNIQUE模式下物理存储由MD5管理，多个文件记录引用同一MD5是正常行为（去重存储）。删除文件记录时仅删除记录本身，不删除物理存储。
- 失效数据记录在被认领后不应立即标记为"已认领"（因为允许多次认领），而是保留为"待处理"状态直到管理员主动确认或所有认领完成后。
- 需要增加"已发布可认领"状态，表示管理员已审核并开放给用户认领。

> 意见：
> 1. 如果UNIQUE模式下物理存储仍有被文件记录指向，那么这个物理存储就不是失效的存储数据。
> 2. 只有存在UNIQUE模式下物理存储没有被任何文件记录指向时，才会被检测为失效数据，并生成失效数据记录供认领。
> 3. 需要增加"已发布可认领"状态，表示管理员已审核并开放给用户认领。

### 9. 丢弃操作的安全边界 [高]

**现状**：README提到"执行数据丢弃：删除物理存储文件（失效物理存储）或删除文件记录（失效文件记录）"。

**问题**：

- UNIQUE模式：丢弃失效物理存储（删除物理文件）时，是否检查该MD5是否还有其他有效文件记录引用？如果有，删除物理文件会导致其他文件记录变成新的失效数据。
- UNIQUE模式：丢弃失效物理存储时，是否检查该文件是否已被认领（已有认领创建的文件记录指向它）？
- RAW模式：丢弃失效文件记录时，是否检查该记录是否已被修复？

**建议**：丢弃操作前必须执行引用检查：

- UNIQUE模式丢弃失效物理存储：检查该MD5是否还有任何有效文件记录（含认领创建的），如有则拒绝删除物理文件，提示管理员先处理相关文件记录。
- UNIQUE模式丢弃失效物理存储：即使没有有效文件记录，如果该失效数据处于"已发布可认领"状态，也应提示管理员先取消发布。
- RAW模式丢弃失效文件记录：检查状态是否为"待处理"，已修复的记录不允许丢弃。

> 意见：
> 1. UNIQUE模式下，存在失效物理存储时，说明该物理存储没有被任何文件记录指向，因此不存在删除物理文件会导致其他文件记录变成新的失效数据的风险。
> 2. 如果文件已被认领过，且系统仍存在该md5的引用，则不允许丢弃。
> 3. RAW模式丢弃失效文件记录：检查状态是否为"待处理"，已修复的记录不允许丢弃。

### 10. 检测任务失败后的数据状态 [中]

**现状**：异步任务支持失败重试。

**问题**：检测任务执行到一半失败，已写入的检测结果如何处理？重试时是重新全量扫描还是断点续扫？

**建议**：检测任务采用"先扫描到内存，最后一次性写入"
的策略。任务失败时不写入任何结果，重试时全量重新扫描。如果数据量过大无法全部放入内存，可采用临时表方案：先写入临时标记，全部完成后再将临时标记转为正式记录。

> 意见：
> 保留检测结果，重试时从头再来。

### 11. 公共资源(uid=0)的写入权限 [中]

**现状**：认领时"公共网盘只有管理员能选择"。

**问题**：

- 普通用户认领失效数据到自己的网盘时，uid应为该用户的uid，这没问题。
- 但如果普通用户认领后，管理员能否将该失效数据的认领路径设为公共网盘(uid=0)？

**建议**：认领到公共网盘的操作仅限管理员，且需要在Controller层通过 `@RolesAllowed(SysRole.ADMIN)` 注解保护。普通用户只能认领到自己的私人网盘。

> 意见：
> 1.
失效数据未发布为公共可认领，或保存到的目标位置为公共网盘，或非自己的个人网盘时，需要校验当前用户（认领人）具有管理员权限。即：普通用户只能认领已发布公共可认领的数据到自己的私人网盘，管理员可以认领任何数据到任何位置（包括公共网盘）。
> 2. 失效的物理存储数据可被多次认领

### 12. 失效数据统计信息 [建议]

检测完成后应提供统计概览：失效数据总数、总占用空间、按类型分布（失效文件记录 vs 失效物理存储）、按用户分布。帮助管理员快速了解数据状况和优先级。

> 意见：本次不处理

### 13. 检测范围控制 [建议]

当前设计是全量扫描所有用户数据。对于大规模部署，应支持：

- 按用户范围检测（仅检测指定用户）
- 按目录范围检测（仅检测指定路径下）

修复/认领/丢弃等操作应记录审计日志，包含操作人、操作时间、操作类型、目标数据、操作结果。便于事后追溯和问题排查。

> 意见：采纳

## 五、建议补充的设计内容

1. **失效数据实体状态机**：
    - RAW模式：`待处理 -> 已修复 / 已丢弃`
    - UNIQUE模式：`待处理 -> 已发布可认领 -> (多次认领) -> 已丢弃`
    - 明确各状态转换条件和允许的操作

> 意见：采纳

2. **认领记录实体**（仅UNIQUE模式）：记录认领用户ID、时间、文件名、路径、关联的失效数据ID

> 意见：采纳

3. **各操作的前置条件检查清单**：
    - 快速修复（RAW）：检查物理存储是否存在、原路径是否可用
    - 认领（UNIQUE）：检查文件名冲突、公共网盘权限
    - 丢弃：检查MD5引用计数、认领状态、修复状态

> 意见：
> 1. RAW模式下的快速修复逻辑：丢失文件记录，但物理存储存在且文件记录的原路径可用时，允许快速修复；如果文件记录原路径不可用，则失败。丢失文件记录且物理存储不存在时，检查能否通过文件记录的md5找到网盘其他位置相同的文件，如果存在则复制一份，否则修复失败。
> 2. 认领逻辑见前面的校验建议，主要是文件名冲突和权限检查。
> 3. 丢弃建议：采纳

4. **存储模式操作矩阵**：

   | 操作 | RAW | UNIQUE |
   |------|-----|--------|
   | 快速修复 | 支持 | 不支持 |
   | 认领 | 不支持 | 支持（多次） |
   | 丢弃（物理存储） | 不适用 | 支持（需检查引用） |
   | 丢弃（文件记录） | 支持 | 不适用 |
   | 文件识别 | 支持 | 支持 |

   > 意见：确认
5. **UNIQUE模式认领路径规则**：明确认领创建的文件记录在UNIQUE存储模式下的路径构建逻辑
   > 意见：确认

## 六、基于审阅意见的后续评审

### 15. 检测失败重试与已有结果的冲突 [中]

**背景**：问题10的审阅意见为"保留检测结果，重试时从头再来"。

**问题**：
- 检测任务执行到一半失败，已扫描并写入了部分失效数据记录（如已发现5条）。
- 管理员选择重试，重试任务从头全量扫描，发现了3条（其中2条与上次重叠，1条是新发现的）。
- 最终结果是保留上次的5条 + 本次的3条 = 8条？还是用本次的3条覆盖上次的5条？
- 如果是覆盖，那"保留检测结果"指的是失败后管理员仍可查看已有结果，但重试时会清除？

**建议**：明确为"重试时清除上一次未完成的检测结果，重新全量扫描"。保留结果的含义是：任务失败后、管理员决定重试之前，已有的部分结果仍可浏览。一旦发起重试，旧的未处理结果会被清除。

> 意见：采纳

### 16. 认领后失效数据记录的状态流转 [高]

**背景**：问题8审阅意见确认"物理存储没有被任何文件记录指向时才会被检测为失效数据"，并增加"已发布可认领"状态。

**问题**：认领操作创建了文件记录后，该物理存储就不再是"无文件记录指向"的状态了。但失效数据记录本身如何变化？

场景推演：
1. 管理员检测到失效物理存储A，生成失效数据记录（状态：待处理）
2. 管理员将A发布为可认领（状态：已发布可认领）
3. 用户X认领A，创建了文件记录（该物理存储不再失效）
4. 此时失效数据记录的状态应变为？是"已认领"还是保留"已发布可认领"（因为允许多次认领）？

**建议**：失效数据记录的状态应独立于物理存储的实际状态。建议增加"已被认领"状态表示至少被认领过一次，但仍然允许继续认领（因为是多次认领机制）。管理员可以手动将状态从"已被认领"变为"已丢弃"来关闭认领通道。或者更简单：保持"已发布可认领"状态不变，直到管理员主动关闭（丢弃/取消发布）。

> 意见：状态改为“已认领”，并通过一个另外的数据表来记录认领的记录。

### 17. 丢弃时的认领状态检查细节 [高]

**背景**：问题9审阅意见"如果文件已被认领过，且系统仍存在该md5的引用，则不允许丢弃"。

**问题**：
- "系统仍存在该md5的引用"指的是什么？是指通过认领创建的文件记录吗？
- 假设用户X认领了文件A，后来用户X自己删除了该文件记录（不是通过丢弃操作，而是正常的文件删除）。此时系统中已无该MD5的引用，管理员能否丢弃该失效物理存储？
- 如果用户X认领了文件A，用户Y也认领了文件A，X删除了自己的文件记录，Y的还在。此时管理员能否丢弃？（应不能，因为Y的引用还在）

**建议**：丢弃失效物理存储前，检查系统中是否存在任何有效的文件记录引用该MD5（包括通过认领创建的文件记录）。只有当MD5引用计数为0时才允许丢弃。这个检查应该实时查询文件记录表，而非仅依赖失效数据记录的状态。

> 意见：建议正确，采纳

### 18. RAW模式快速修复 - MD5查找复制的边界 [中]

**背景**：问题3的审阅意见补充了RAW模式修复逻辑："丢失文件记录且物理存储不存在时，检查能否通过文件记录的md5找到网盘其他位置相同的文件，如果存在则复制一份，否则修复失败。"

**问题**：
- "找到网盘其他位置相同的文件"的搜索范围是什么？是仅搜索当前用户的文件，还是搜索所有用户的文件（包括公共网盘）？
- 如果找到了其他用户的文件，复制操作是否需要目标用户的权限？（读取其他用户的文件内容是否需要特殊权限？）
- UNIQUE模式下的物理存储天然就是按MD5去重的，但RAW模式是否也存在MD5去重机制？如果RAW模式不做去重，那"其他位置相同的文件"的可能性有多大？

**建议**：明确搜索范围为所有用户和公共网盘（因为修复的是文件记录，最终恢复到原用户/原路径，读取其他用户的文件内容用于恢复是合理的管理员操作）。如果涉及权限问题，可在修复服务层使用管理员上下文执行。

> 意见：可在所有用户和公共网盘下查找相同md5的文件记录，如果找到则复制一份到物理存储路径，否则修复失败。

### 19. 认领权限校验的完整规则 [中]

**背景**：问题11审阅意见明确了权限规则。

**问题**：需要将权限规则转化为可执行的校验逻辑，当前描述存在一个边界场景：
- 普通用户认领已发布公共可认领的数据，保存到自己的私人网盘 -> 允许 ✓
- 管理员认领任何数据到任何位置 -> 允许 ✓
- 普通用户认领**未发布**公共可认领的数据 -> 不允许 ✓
- 普通用户认领已发布公共可认领的数据，保存到**公共网盘** -> 不允许（需管理员） ✓
- 普通用户认领已发布公共可认领的数据，保存到**其他用户的网盘** -> 不允许？

**建议**：补充明确普通用户只能认领到自己的网盘（targetUid == 当前用户uid），不能认领到其他用户的网盘。校验规则总结：
- 失效数据未发布为公共可认领：仅管理员可操作
- 失效数据已发布为公共可认领：任何用户可认领，但targetUid必须为当前用户uid；管理员无此限制
- targetUid为0（公共网盘）：仅管理员可操作

> 意见：采纳，与用户上传/保存文件的校验逻辑一致，普通用户只允许在自己的私人网盘操作。可直接使用UIDValidator校验。

## 七、第三轮评审 - 细节闭环检查

### 20. UNIQUE模式"已认领"状态下是否仍允许多次认领 [高]

**背景**：问题16审阅意见"状态改为已认领，并通过另外的数据表来记录认领的记录"。结合问题11"失效的物理存储数据可被多次认领"。

**问题**：状态变为"已认领"后，是否仍允许其他用户继续认领？
- 如果允许：状态"已认领"仅表示"至少被认领过一次"，不阻止后续认领。此时需要一个额外的状态（如"已关闭认领"）来由管理员主动关闭认领通道。
- 如果不允许：则违背了"可被多次认领"的设计目标。

**建议**：确认"已认领"状态下仍允许认领。增加"已关闭认领"状态（或复用"已丢弃"），由管理员主动关闭。最终状态机：
`待处理 -> 已发布可认领 -> 已认领(可继续认领) -> 已关闭认领/已丢弃`

> 意见：
> 认领后状态更新为“已认领”，但不影响后续认领。管理员后续可手动将该失效数据标记为处理完成。处理完成后无法再继续认领或丢弃。

### 21. UNIQUE模式缺少"取消发布"状态转换 [中]

**背景**：状态机定义为 `待处理 -> 已发布可认领 -> 已认领 -> 已丢弃`。

**问题**：管理员将失效数据发布为"已发布可认领"后，如果想撤回发布（例如发现数据有误、不应公开），当前状态机没有回到"待处理"的路径。

**建议**：增加"已发布可认领 -> 待处理"的取消发布操作。取消发布时需检查是否已有认领记录：
- 无认领记录：直接回到"待处理"
- 有认领记录：不允许取消发布（已有用户依赖该数据），提示管理员先处理认领记录

> 意见：采纳建议

### 22. 丢弃UNIQUE失效物理存储的完整前置条件 [中]

**背景**：问题17采纳"实时查询文件记录表，MD5引用计数为0时才允许丢弃"。

**问题**：结合状态机，丢弃UNIQUE失效物理存储的完整前置条件应为：
1. 该MD5在文件记录表中无任何引用（含认领创建的记录）
2. 失效数据记录状态不为"已发布可认领"（防止丢弃后用户仍能看到可认领的数据）

但存在边界场景：如果状态为"已认领"，说明曾有认领记录，但用户可能已自行删除了所有认领的文件记录（引用计数归零）。此时能否丢弃？

**建议**：可以丢弃。丢弃操作只检查实时引用计数，不依赖失效数据记录的状态。如果引用计数为0，即使状态为"已认领"（历史曾有认领），也可以丢弃。丢弃时将失效数据记录状态标记为"已丢弃"。

> 意见：简化流程，已认领了不允许丢弃。

### 23. 认领记录表的查询场景 [低]

**背景**：问题16审阅意见确认"通过另外的数据表来记录认领的记录"。

**问题**：认领记录表的主要查询场景是什么？这会影响表的设计和索引：
- 管理员查看某条失效数据的所有认领记录（按失效数据ID查询）
- 某用户查看自己的所有认领记录（按用户ID查询）
- 系统检查某失效数据是否可丢弃（检查是否有认领记录 + 检查引用计数）

**建议**：认领记录表至少需要以下字段和索引：
- 字段：id, invalidDataId, userId(targetUid), fileName, savePath, claimTime
- 索引：invalidDataId（管理员查看）、userId（用户查看自己的认领记录）

> 意见：
> - 用户每认领一次就添加一条记录
> - 继承 AuditModel，添加以下字段：invalid_data_id、target_uid、file_name、save_path。(target_uid表示保存的目标网盘id，0是公共网盘，>0表示对应用户id的私人网盘，AuditModel的uid字段表示操作人用户id)。

## 八、第四轮评审 - 状态机完整性检查

### 24. UNIQUE模式待处理状态缺少直接丢弃操作 [中]

**背景**：根据前几轮审阅，UNIQUE模式状态机为：
`待处理 -> 已发布可认领 -> 已认领 -> 处理完成`
以及 `待处理 -> 已丢弃`（通过取消发布后）。

**问题**：管理员检测到失效物理存储后，如果明确不需要认领（如垃圾文件、临时文件），能否直接丢弃而无需先发布？
- 当前设计中，待处理状态的操作只有"发布为可认领"
- 如果不允许直接丢弃，管理员必须走"发布 -> 发现无人认领 -> 取消发布 -> 丢弃"的流程，过于繁琐

**建议**：待处理状态增加"直接丢弃"操作。丢弃前检查该MD5引用计数为0（这是检测的前提条件，应天然满足）。

> 意见：采纳建议

## 九、第五轮评审 - 最终闭环确认

基于第四轮审阅意见，重新梳理完整的状态机和操作矩阵，确认是否闭环。

### 28. 最终状态机确认

**UNIQUE模式状态机：**

```
待处理 --发布--> 已发布可认领 --认领--> 已认领 --标记完成--> 处理完成
  |                 |                    |
  |                 | (取消发布)          |
  |<----------------|                    |
  |                                      |
  +---丢弃-----------------------------> 处理完成(丢弃)
```

**RAW模式状态机：**

```
待处理 --快速修复成功--> 处理完成(自动修复)
  |
  +---丢弃-----------> 处理完成(丢弃)
```

**处理完成的触发条件：**
- UNIQUE模式：已被认领 + 管理员主动标记处理完成（关闭认领通道）
- UNIQUE模式：直接丢弃（引用计数=0时）
- RAW模式：快速修复成功

**处理方式字段值：**
- `丢弃`：删除了失效数据（物理存储文件或失效文件记录）
- `认领`：数据已被认领，管理员关闭认领通道
- `自动修复`：RAW模式快速修复成功

**结论**：状态机已闭环。"处理完成"作为唯一终态，通过`处理方式`字段区分不同的完成原因。

### 29. 检测结果清理策略确认 [中]

**背景**：问题1的审阅意见为"全量替换，清除上一轮未处理的检测结果"。问题28确认"处理完成"为终态。

**问题**：检测清理策略是否需要区分"处理完成"和"待处理"？
- 如果全量清除（包括"处理完成"的记录），会丢失操作历史
- 如果只清除"待处理"记录，保留"处理完成"记录，管理员可以看到历史处理记录

**建议**：检测开始时只清除上一轮"待处理"状态的记录，保留所有"处理完成"的记录作为历史。这样管理员既能看到最新检测结果，也能看到历史处理记录。

> 意见：采纳建议

### 30. 最终操作矩阵确认

|------|-----|--------|

| 丢弃（文件记录） | 支持 | 不适用 |
