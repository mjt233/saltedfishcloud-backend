# 失效数据检测策略

## 1. 物理存储与文件记录扫描

- 通过`storeServiceFactory.getStoreService().getStorageProvider()`获取存储服务底层的存储提供者`Storage`，基于`Storage`的操作对物理存储数据进行检查。
- 通过`fileSystemMetadataOperator`对所有用户的文件记录进行扫描，检查文件记录的有效性。

数据比对：
- 当通过`Storage`物理存储中的数据未存在能被`fileSystemMetadataOperator`有效关联时，说明该文件为失效数据；
- 当通过`fileSystemMetadataOperator`扫描到的文件记录在物理存储中未能被`Storage`扫描到时，说明该文件记录为失效数据。

### Storage的文件列表

需要根据不同的存储模式，设计不同的物理存储路径检查策略

存储模式可通过`sysCommonConfig.getStoreMode()`获取

#### RAW 模式

- 公共网盘（uid=0）： 在`storeServiceFactory.getStoreService().getPublicRoot()`下获取文件列表
- 用户网盘（uid>0）：在{`storeServiceFactory.getStoreService().getStoreRoot`}/user_file/用户id 下获取文件列表

在 publicRoot 或 storeRoot/user_file/用户id 下的文件的物理存储路径和文件名即为该文件的网盘路径和文件名，属于可直接识别的文件。

#### UNIQUE 模式

UNIQUE模式不区分公共网盘和用户网盘，所有文件都存储在`storeServiceFactory.getStoreService().getStoreRoot()`的多层子目录下，所有无拓展名的文件即为物理存储文件。

这些无拓展名的文件名本身即为该文件的md5值。

由于文件仅通过md5值进行存储，无法直接从物理存储路径中识别出文件的网盘路径和文件名，因此属于待识别文件。

## 2. 记录失效数据

失效数据明细需要包含以下字段：

- 失效数据类型（失效文件记录/失效物理存储）
- 完整物理存储路径（对于失效物理存储，表示现存文件的物理存储路径；对于失效文件记录，表示记录指向的物理存储路径）
- 所属用户id（公共网盘文件为0，用户网盘文件>0）
- 网盘路径（对于失效的UNIQUE模式的物理存储，由于无法识别的网盘路径，值为null）
- 文件大小
- 最后修改时间
- 是否为待识别的文件（UNIQUE模式下为无法直接识别的文件，RAW模式无需识别，直接根据拓展名识别即可）
- 文件类型（基于系统支持的文件类型识别的typeId，表示图片、视频、文档、文本等数据，该字段为null表示未知/待识别）
- 元数据（仅UNIQUE模式下对该文件进行识别操作后产生）
- 状态（见状态机定义）
- 处理方式（仅"处理完成"状态时有值，见处理方式定义）

## 3. 失效数据状态机

### 终态说明

"处理完成"作为唯一终态，通过`处理方式`字段区分不同的完成原因。

处理方式字段值：
- `丢弃`：删除了失效数据（物理存储文件或失效文件记录）
- `认领`：UNIQUE模式数据已被认领，管理员关闭认领通道
- `自动修复`：RAW模式快速修复成功

### UNIQUE模式状态机

```
待处理 --发布--> 已发布可认领 --认领--> 已认领 --标记完成--> 处理完成
  |                 |                    |
  |                 | (取消发布)          |
  |<----------------|                    |
  |                                      |
  +---丢弃-----------------------------> 处理完成(丢弃)
```

状态转换条件：
- 待处理 -> 已发布可认领：管理员发布为公开可认领
- 已发布可认领 -> 已认领：用户认领成功（允许多次认领，状态仅变化一次）
- 已认领 -> 处理完成：管理员手动标记处理完成（关闭认领通道）
- 已发布可认领 -> 待处理：管理员取消发布（需无认领记录）
- 待处理 -> 处理完成(丢弃)：管理员直接丢弃（引用计数=0时）

### RAW模式状态机

```
待处理 --快速修复成功--> 处理完成(自动修复)
  |
  +---丢弃-----------> 处理完成(丢弃)
```

状态转换条件：
- 待处理 -> 处理完成(自动修复)：快速修复成功
- 待处理 -> 处理完成(丢弃)：管理员丢弃

## 4. 检测执行策略

### 并发控制

限制只能同时有一个进行中的检测或文件类型识别任务，检测或识别过程中不允许再次发起检测或文件类型识别任务。

### 检测结果清理策略

检测开始时只清除上一轮"待处理"状态的记录，保留所有"处理完成"的记录作为历史。这样管理员既能看到最新检测结果，也能看到历史处理记录。

### 任务失败与重试

- 任务失败后保留已有的检测结果，管理员可浏览
- 管理员发起重试时，清除上一次未完成的检测结果，重新全量扫描

## 5. 认领记录表（仅UNIQUE模式）

每次认领创建一条记录，继承AuditModel（AuditModel的uid字段表示操作人用户id）。

额外字段：
- `invalid_data_id`：关联的失效数据记录ID
- `target_uid`：保存的目标网盘id（0=公共网盘，>0=对应用户id的私人网盘）
- `file_name`：认领时填写的文件名
- `save_path`：认领时填写的保存路径

索引：
- `idx_claim_record_invalid_data_id`（管理员查看某条失效数据的所有认领记录）
- `idx_claim_record_target_uid`（用户查看自己的认领记录）

## 6. 操作前置条件

### 快速修复（RAW模式）

**失效文件记录修复：**
1. 检查物理存储是否存在（通过md5查找）
2. 如果物理存储存在且原路径可用，重建文件记录
3. 如果物理存储存在但原路径不可用，修复失败（路径冲突）
4. 如果物理存储不存在，在所有用户和公共网盘下查找相同md5的文件记录，如果找到则复制一份到物理存储路径，否则修复失败

**失效物理存储修复：**
重建该存储对应的文件记录，恢复文件访问路径和相关信息。

### 认领（UNIQUE模式）

权限校验（使用UIDValidator）：
- 失效数据未发布为公共可认领：仅管理员可操作
- 失效数据已发布为公共可认领：任何用户可认领，但targetUid必须为当前用户uid；管理员无此限制
- targetUid为0（公共网盘）：仅管理员可操作

业务校验：
- 文件名+路径冲突检查：拒绝操作并提示用户文件名已存在
- 状态校验：仅"已发布可认领"和"已认领"状态允许认领

### 丢弃

**UNIQUE模式丢弃失效物理存储：**
- 前置条件：MD5引用计数为0（实时查询文件记录表）且状态不为"已认领"
- 已认领的失效数据不允许丢弃

**RAW模式丢弃失效文件记录：**
- 前置条件：状态为"待处理"，已处理完成的记录不允许丢弃

### 标记处理完成

- UNIQUE模式：仅"已认领"状态允许标记处理完成
- RAW模式：快速修复成功时自动标记

## 7. 存储模式操作矩阵

| 操作 | RAW | UNIQUE |
|------|-----|--------|
| 快速修复 | 支持 | 不支持 |
| 认领 | 不支持 | 支持（多次） |
| 发布为可认领 | 不适用 | 支持 |
| 取消发布 | 不适用 | 支持（无认领记录时） |
| 丢弃（物理存储） | 不适用 | 支持（引用计数=0且未被认领） |
| 丢弃（文件记录） | 支持 | 不适用 |
| 标记处理完成 | 修复成功时自动 | 管理员手动 |
| 文件识别 | 支持 | 支持 |
